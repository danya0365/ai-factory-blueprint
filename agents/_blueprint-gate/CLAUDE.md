## หน้าที่

เป็น **gate** คั่นกลางสายพาน: ตรวจงานจาก upstream ตามเกณฑ์ ถ้า**ผ่าน**ปล่อยไหลต่อตามปกติ
ถ้า**ไม่ผ่าน** ตีงานกลับไปแก้ที่ stage ก่อนหน้า (reject loop) แทนที่จะปล่อยของเสียไหลลงปลายน้ำ

## Input — อ่านจากไหน
- อ่าน `../<UPSTREAM>/output/<JOB_ID>.json` (แทน `<UPSTREAM>` ด้วยขั้นก่อนหน้าจริงตอน wire เข้าสาย)
- **อ่านอย่างเดียว — ห้ามแก้ไฟล์ upstream**

## Output — เขียนไปไหน
- เขียน `output/<JOB_ID>.json` ในโฟลเดอร์ของ agent นี้เท่านั้น

## สิ่งที่ต้องทำ
1. คัดลอก envelope จาก upstream มาเป็นฐาน (คง `job_id` เดิม)
2. ตรวจตามเกณฑ์ที่กำหนด (เช่น `valid_count` ต้องเท่ากับ `record_count`, ห้ามมี issue ระดับ `error`)
3. **ถ้าผ่าน:**
   - `status` = `"ok"`, **ไม่ต้องใส่** `reject_to`
   - ต่อ `history` เช่น `"gate: passed (2/2 valid)"`
4. **ถ้าไม่ผ่าน:**
   - `status` = `"rejected"`
   - `reject_to` = ชื่อ stage ที่จะให้วนกลับไปแก้ (เช่น `"02-refine"`) — ต้องเป็นขั้น**ก่อนหน้า**ในสาย
   - บันทึกเหตุผลที่ตีกลับลง `issues[]` ให้ชัด (record ไหน field ไหน ทำไมไม่ผ่าน) — ขั้นปลายทางจะอ่านไปแก้
   - ต่อ `history` เช่น `"gate: rejected -> 02-refine (1 record invalid email)"`
5. ตั้ง `stage`/`produced_by` = ชื่อโฟลเดอร์จริงของ gate, อัปเดต `produced_at`, คง `job_id` เดิม
6. **อย่าแตะ `attempts`** — orchestrator เป็นคนเพิ่มค่าเอง
7. เขียนลง `output/<JOB_ID>.json` (indent 2)

## เสร็จแล้ว
รายงานสั้นๆ ว่า ผ่าน หรือ ตีกลับไป stage ไหน เพราะอะไร แล้วจบงาน
